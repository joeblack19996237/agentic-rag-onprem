# 06 — API Contracts

> Stage 7 (`spec-writer`) deliverable. Expands the DEC-107 stub into the full canonical API contract. Source of truth for *behavior* (what each endpoint does, why) remains `04-architecture.md` §7 and the cross-referenced sections noted per endpoint — this file owns *contract shape*: schemas, error codes, auth, pagination, idempotency, versioning.

## Plain-English Summary

Four surfaces: Query (the hot path — ask a question, get an answer or an honest refusal), Ingest (upload a document, track its indexing progress), Admin (document/ACL/threshold/model/CDC management, audit pull), and Operational (health check). Every endpoint declares its own auth, every error has a typed shape, and refusals are never HTTP errors — they're 200 responses with a typed reason, because refusing to answer is a product feature, not a failure (DEC-042).

## API Style

REST over HTTPS, JSON request/response bodies, OpenAPI 3.x auto-generated by FastAPI (DEC-033). No GraphQL, no RPC framework, no MCP tool surface in MVP (`21-tools-and-mcp.md` is skipped per `00-index.md`'s Stage 6 skip rationale — MVP has no agent tool surface, DEC-039).

## Authentication and Authorization

| Mechanism | Applies to | Notes |
|---|---|---|
| JWT bearer (RS256, ES256, or EdDSA only — `HS*` and `none` rejected) | All authenticated endpoints | DEC-061; JWKS via `OIDCAdapter` (online) or static pre-imported bundle (air-gap, DEC-062) |
| Admin API key | Admin-surface endpoints, as an alternative to JWT | NFR-009 |
| Unauthenticated | `GET /ready` only | The sole exception, by design — install verification and widget load-gating must work before any token exists |

Every endpoint below states its auth explicitly; there is no "default auth" a reader has to infer.

## Versioning

- URL-path versioning: `/v1/...`. A breaking change to any endpoint's request/response shape ships as `/v2/...` alongside the still-supported `/v1/...`, not an in-place breaking change
- **Deprecation window**: a minimum of 90 days between announcing a `/v1` endpoint's deprecation and removing it, published in release notes — chosen to match the LCC annual-support cadence (DEC-028) so a vendor integrator on an annual support contract always has at least one support cycle of notice
- OpenAPI schema version is pinned per release; vendor integrators can diff schema versions to detect contract drift before upgrading

## Idempotency

| Endpoint class | Idempotency mechanism |
|---|---|
| `POST /v1/ingest` | Not idempotent by document content — re-uploading the same file produces a new `document_id`. Idempotency at this layer is the caller's responsibility (track your own `document_id` after first upload) |
| `POST /v1/query` | Not idempotent — each call is a new turn, even with the same query text and `conversation_id` (the answer-cache, `04-architecture.md` §7B.5, may return a cached result, but this is a performance optimization the caller cannot rely on as a correctness guarantee — cache keys expire and invalidate) |
| CDC webhook receiver (internal, not a public API endpoint but documented here for completeness) | Idempotent via `(event_id, source_timestamp)` dedup key, 7-day window (RC-T8-01) — the ECM may safely redeliver a webhook |
| `POST /v1/admin/eval` | Each call starts a new run (`eval_runs.run_id`); not idempotent, but safe to call concurrently — see Admin API section for the sync/async resolution |

## Pagination

- **Cursor-based**, not offset-based, on every list/query endpoint — offset pagination degrades under concurrent inserts (a common failure mode on `audit_events`, which is append-only and constantly growing)
- Cursor shape: opaque base64-encoded string containing `{last_seen_timestamp, last_seen_id}`; clients must treat it as opaque and never construct one manually
- Every paginated response includes `next_cursor: string | null` (`null` means no more pages)

## Rate Limits

Per NFR-017: default 60 queries per token per minute, burst 10 in 5 seconds, admin-configurable. Per-IP secondary limit for the unauthenticated demo widget path. Over-limit responses use the standard error shape below with HTTP 429 and a `Retry-After` header.

## Error Schema

All non-2xx responses share one shape:

```json
{
  "error": {
    "code": "string (machine-readable, stable across versions)",
    "message": "string (human-readable, safe to display)",
    "request_id": "string (correlates to a trace span, §12.3 OTel conventions)",
    "details": {}
  }
}
```

**Refusals are not errors** (DEC-042) — a refusal is a `200` with `{answer: null, refusal_reason: RefusalClass, ...}`, never routed through this error shape. This distinction is deliberate and load-bearing for the product's positioning: an error means the system failed to do its job; a refusal means the system did its job (declining to guess).

| HTTP status | `error.code` | When |
|---|---|---|
| 400 | `invalid_request` | Malformed request body, missing required field |
| 401 | `unauthenticated` | Missing or invalid bearer token / API key |
| 403 | `forbidden` | Valid token, insufficient scope (e.g. non-admin token calling an admin endpoint) |
| 404 | `not_found` | Referenced `document_id`/`audit_id`/etc. does not exist |
| 409 | `conflict` | e.g. a second model-swap request while one is already in progress for the same role (`07-database.md`'s partial-unique-active-row constraint surfaces here) |
| 415 | `unsupported_media_type` | Ingest upload in an unsupported format (REQ-001) |
| 429 | `rate_limited` | NFR-017 rate limit exceeded |
| 500 | `internal_error` | Unhandled exception; audit row is still written before this response is sent (§8.4) |
| 503 | `service_unavailable` | A required backend (vLLM, TEI, etc.) is unhealthy (`20-agent-behavior.md` §6) |
| 504 | `generation_timeout` | Generation exceeded its time budget; client may retry with the same `conversation_id` |

## 1. Query API (the hot path)

| Contract ID | Type | Name | Caller | Callee | Auth | Input | Output | Errors |
|---|---|---|---|---|---|---|---|---|
| API-Q-01 | REST | `POST /v1/query` | Widget / vendor integration | `api/` | JWT bearer or admin API key | See schema below | See schema below | 400, 401, 403, 429, 500, 503, 504 |

### Request Schema

```json
{
  "query": "string, required, 1-4000 chars",
  "conversation_id": "string, optional (UUID) — omit to start a new conversation"
}
```

### Response Schema — Answered

```json
{
  "answer": "string",
  "citations": [
    {
      "chunk_id": "string",
      "document_id": "string",
      "page_number": "integer | null",
      "span_offset": {"start": "integer", "end": "integer"},
      "cited_span_text": "string — verbatim snapshot, DEC-087",
      "nli_score": "float 0..1"
    }
  ],
  "refusal_reason": null,
  "audit_id": "string (UUID)",
  "conversation_id": "string (UUID) — echoed or newly generated",
  "latency_ms": "integer"
}
```

### Response Schema — Refused

```json
{
  "answer": null,
  "citations": [],
  "refusal_reason": "no_recall | low_grounding | access_denied | policy_blocked | verification_unavailable",
  "suggested_documents": [
    {"document_id": "string", "title": "string"}
  ],
  "audit_id": "string (UUID)",
  "conversation_id": "string (UUID)",
  "latency_ms": "integer"
}
```

`suggested_documents` is populated only for `no_recall` (up to 3 ACL-filtered neighboring docs, REQ-006a); empty array for the other 4 refusal classes.

### Request/Response Examples

**Example 1 — Answered, single citation**

Request:
```json
{"query": "What is the standard notice period for a billing dispute?", "conversation_id": "c-8f21"}
```

Response (200):
```json
{
  "answer": "The standard notice period is 30 days from the date of the billing statement [1].",
  "citations": [{"chunk_id": "chk-441", "document_id": "doc-102", "page_number": 3, "span_offset": {"start": 220, "end": 340}, "cited_span_text": "Customers must be notified of any billing dispute within thirty (30) days of the statement date.", "nli_score": 0.94}],
  "refusal_reason": null,
  "audit_id": "a1b2c3d4-...",
  "conversation_id": "c-8f21",
  "latency_ms": 6420
}
```

**Example 2 — Refused, `no_recall`**

Response (200):
```json
{
  "answer": null,
  "citations": [],
  "refusal_reason": "no_recall",
  "suggested_documents": [{"document_id": "doc-201", "title": "Billing Adjustment Policy v3"}],
  "audit_id": "e5f6a7b8-...",
  "conversation_id": "c-8f21",
  "latency_ms": 890
}
```

**Example 3 — Refused, `access_denied`, transparent mode**

Response (200):
```json
{"answer": null, "citations": [], "refusal_reason": "access_denied", "suggested_documents": [], "audit_id": "...", "conversation_id": "c-8f21", "latency_ms": 340}
```

Under `opaque` mode (DEC-069), the same underlying event returns `refusal_reason: "no_recall"` to this caller — the audit record alone (`audit_events.refusal_reason_actual`) retains `access_denied`.

### Validation Rules

- `query` must be non-empty after trimming whitespace; server rejects with `400 invalid_request` otherwise
- `conversation_id`, if supplied, must be a syntactically valid UUID; an unrecognized (never-seen) UUID is accepted and treated as the start of a new conversation under that ID — not an error, since a client-generated ID is a reasonable pattern for pre-registering a conversation

### Security and Abuse Handling

- Client-supplied conversation history is never accepted or trusted — only `query` and `conversation_id` are read; the server reconstructs history server-side from `audit_events` (§2.4). There is no request field for "prior messages," by design — this closes the fabricated-history prompt-injection vector at the schema level, not just the prompt-construction level
- Rate limiting per NFR-017 applies per-token; the unauthenticated demo-widget path (if enabled) has a secondary per-IP limit

## 2. Ingest API

| Contract ID | Type | Name | Caller | Callee | Auth | Input | Output | Errors |
|---|---|---|---|---|---|---|---|---|
| API-I-01 | REST | `POST /v1/ingest` | Admin / vendor automated sync | `ingest/` | JWT bearer (admin scope) or admin API key | Multipart upload | `{document_id, status_url}` | 400, 401, 403, 415, 500 |
| API-I-02 | REST | `GET /v1/ingest/{document_id}` | Admin / vendor automated sync | `ingest/` | Same as above | Path param | Status object (below) | 401, 403, 404 |

### `POST /v1/ingest` Request

Multipart form with one file field, name `file`. Accepted MIME types: `application/pdf` (text-extractable only), `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `text/markdown`, `text/plain`. Optional form fields: `repository_id` (defaults to a single-repository install's default), `acl_override` (advanced; normally Layer 1 ACL comes from `ECMAdapter.get_effective_acl()`, not the upload call).

### `POST /v1/ingest` Response

```json
{"document_id": "string (UUID)", "status_url": "/v1/ingest/{document_id}"}
```

Returned within 1 second of upload (REQ-001 acceptance criterion) — this response precedes parsing, it does not wait for it.

### `GET /v1/ingest/{document_id}` Response

```json
{
  "status": "pending | parsing | indexing | ready | failed",
  "progress": "float 0.0-1.0",
  "errors": ["string array, empty unless status=failed"]
}
```

### Examples

Poll sequence for a 100-page PDF: `{"status": "parsing", "progress": 0.1, "errors": []}` → `{"status": "indexing", "progress": 0.7, "errors": []}` → `{"status": "ready", "progress": 1.0, "errors": []}`, total elapsed ≤ 60s on reference hardware.

Failure example: `{"status": "failed", "progress": 0.3, "errors": ["Unstructured.io parse failure at page 42: malformed PDF structure. PyMuPDF rescue also failed. Retry after re-exporting the source document, or contact support with this document_id."]}`

### Validation Rules

- Format rejected at upload time (before a `document_id` is even issued) if the MIME type is not in the accepted list — `415` with the supported-format list in `error.details.supported_formats`
- `GET .../{document_id}` for an unknown ID returns `404 not_found`, not an empty/zero-valued status object

## 3. Admin API

| Contract ID | Type | Name | Caller | Callee | Auth | Input | Output | Errors |
|---|---|---|---|---|---|---|---|---|
| API-A-01 | REST | `GET /v1/admin/documents` | Admin | `admin/` | Admin API key or JWT (admin scope) | Query params: `cursor?`, `limit? (default 50, max 200)` | Paginated document list | 401, 403 |
| API-A-02 | REST | `PUT /v1/admin/documents/{document_id}` | Admin | `admin/` | Same | `{acl?: {...}, lifecycle_state?: "deleted", authority_state?: "authoritative"\|"draft"\|"deprecated"}` | Updated document object | 400, 401, 403, 404 |
| API-A-03 | REST | `GET /v1/admin/audit` | Admin | `audit/` (read-only) | Same | Query params: `from`, `to`, `user_id?`, `cursor?`, `limit?` | Paginated audit records | 400, 401, 403 |
| API-A-04 | REST | `GET /v1/admin/audit/events` | Vendor SIEM | `audit/` (read-only) | Same | Query params: `from`, `to`, `user_id?`, `cursor?` | NDJSON stream, cursor-paginated (REQ-043) | 400, 401, 403 |
| API-A-05 | REST | `PUT /v1/admin/config/thresholds` | Admin | `config/` | Same | `{nli_entailment_threshold?, nli_fail_rate_threshold?, retrieval_top1_threshold?}` | Updated config, effective immediately (NFR-010) | 400, 401, 403 |
| API-A-06 | REST | `GET /v1/admin/config/models` | Admin | `config/` | Same | — | Active `model_versions` per role | 401, 403 |
| API-A-07 | REST | `PUT /v1/admin/config/models` | Admin | `config/` | Same (V2) | `{role, adapter_name, model_version}` | Swap initiated; see idempotency note | 400, 401, 403, 409 |
| API-A-08 | REST | `POST /v1/admin/eval` | Admin | `eval/` | Same | `{suite: "golden-smoke" \| "golden-full" \| "customer-*"}` | `{run_id, status_url}` — **async, resolves the stub's open question**: see rationale below | 400, 401, 403 |
| API-A-09 | REST | `GET /v1/admin/eval/{run_id}` | Admin | `eval/` | Same | Path param | Run status + result once complete | 401, 403, 404 |
| API-A-10 | REST | `POST /v1/admin/acl/refresh_user/{user_id}` | Admin | `acl/` | Same | Path param | `{refreshed: true}` | 401, 403, 404 |
| API-A-11 | REST | `POST /v1/admin/acl/refresh_doc/{doc_id}` | Admin | `acl/` | Same | Path param | `{refreshed: true}` | 401, 403, 404 |
| API-A-12 | REST | `PUT /v1/admin/config/cdc` | Admin | `cdc/` | Same | `{cdc_transport_mode: "webhook"\|"poll_only", poll_interval_minutes?}` | Updated CDC config (DEC-102) | 400, 401, 403 |

### Resolving the Stub's Open Question — `POST /v1/admin/eval` sync vs async

**Decision: async, `status_url` pattern, matching `POST /v1/ingest`.** Rationale: the full golden-set ring (150-200 prompts, DEC-078) takes materially longer than a synchronous HTTP request should reasonably block on — running RAGAS metrics + NLI scoring across 150-200 prompts is a multi-minute operation, not a sub-second one. Even the 50-prompt smoke ring is documented to target "≤ 5 min" (REQ-049), which is already too long for a blocking HTTP call given typical load-balancer/proxy timeout defaults (commonly 30-60s). This is a workflow-shape decision consistent with existing product patterns (`POST /v1/ingest` already uses this shape for a similarly long-running operation), not a new architectural commitment.

### Request/Response Examples

**`PUT /v1/admin/config/thresholds`**

Request:
```json
{"nli_entailment_threshold": 0.45}
```

Response (200):
```json
{"nli_entailment_threshold": 0.45, "nli_fail_rate_threshold": 0.30, "retrieval_top1_threshold": 0.62, "effective_immediately": true}
```

**`GET /v1/admin/audit/events` (NDJSON, cursor-paginated)**

Response body (NDJSON, one JSON object per line):
```
{"audit_id": "...", "conversation_id": "...", "refusal_reason_actual": null, "timestamp": "2026-07-06T09:00:00Z", ...}
{"audit_id": "...", "conversation_id": "...", "refusal_reason_actual": "no_recall", "timestamp": "2026-07-06T09:01:12Z", ...}
```
Response header: `X-Next-Cursor: <opaque cursor>` (NDJSON streaming responses carry the cursor in a header rather than a trailing JSON object, since the body itself is a line-delimited stream, not a single JSON document)

### Validation Rules

- `PUT /v1/admin/config/thresholds`: `nli_entailment_threshold` and `nli_fail_rate_threshold` must be in `[0.0, 1.0]`; out-of-range values return `400`
- `PUT /v1/admin/config/cdc`: `poll_interval_minutes` required and must be `> 0` when `cdc_transport_mode = "poll_only"`; ignored (may be omitted) when `cdc_transport_mode = "webhook"`

### Backward Compatibility Rules

- Adding an optional request field or an additional response field is **not** a breaking change and ships within `/v1`
- Removing a field, changing a field's type, or changing an enum's accepted values (e.g. adding a 6th refusal class) **is** breaking and requires `/v2`
- The 5-class `RefusalClass` enum (DEC-042) is treated as closed for `/v1`'s lifetime — any 6th class is a `/v2` concern, not an in-place addition, because vendor integrators' widget code pattern-matches on this enum

## 4. Operational Surface

| Contract ID | Type | Name | Caller | Callee | Auth | Input | Output | Errors |
|---|---|---|---|---|---|---|---|---|
| API-O-01 | REST | `GET /ready` | Widget, install verification, orchestrator health probe | `api/` (aggregates all backends) | None (unauthenticated by design) | — | `{ready: bool, services: {...}}` | — (this endpoint has no error responses; it always returns 200 with `ready: true\|false` in the body, since a health-check endpoint that itself returns 5xx during a startup window would be indistinguishable from "the endpoint is broken" vs "the system is still starting") |

### `GET /ready` Response Schema (DEC-117 full-dependency check)

```json
{
  "ready": "boolean — true only if every service below reports healthy",
  "services": {
    "vllm": "boolean",
    "tei_embed": "boolean",
    "tei_rerank": "boolean",
    "nli": "boolean",
    "safety_input": "boolean",
    "safety_output": "boolean",
    "policy": "boolean",
    "qdrant": "boolean",
    "postgres": "boolean",
    "redis": "boolean"
  }
}
```

### Example

Mid-startup (models still loading):
```json
{"ready": false, "services": {"vllm": false, "tei_embed": true, "tei_rerank": true, "nli": true, "safety_input": false, "safety_output": false, "policy": true, "qdrant": true, "postgres": true, "redis": true}}
```

Fully ready:
```json
{"ready": true, "services": {"vllm": true, "tei_embed": true, "tei_rerank": true, "nli": true, "safety_input": true, "safety_output": true, "policy": true, "qdrant": true, "postgres": true, "redis": true}}
```

The per-service breakdown (not just a top-level boolean) is deliberate: it lets an install-verification script or ops dashboard show *which* service is still starting, rather than an opaque "not ready" with no diagnostic value — directly useful during the REQ-011 ≤30-minute install window.

## Contract Tests

| Test | What it verifies |
|---|---|
| Refusal-as-200 invariant | Every one of the 5 refusal classes, when triggered via a seeded test scenario, returns HTTP 200 with the correct `refusal_reason` — never a 4xx/5xx |
| Client-history rejection | A request body containing an unexpected `history` field is ignored (not merely accepted-and-unused, but verified to have zero influence on the server-reconstructed context) |
| `/ready` aggregation correctness | Stub one backend service as unhealthy → `/ready` returns `ready: false` with that service's boolean `false` and all others accurately reported |
| Pagination cursor stability under concurrent writes | Paginate `GET /v1/admin/audit` while `audit_events` receives concurrent inserts → no record is skipped or duplicated across pages |
| CDC webhook idempotency | Redeliver the same webhook event twice within the 7-day dedup window → second delivery is a no-op, verified via `job_queue`/CDC processing logs showing exactly one state change |
| OpenAPI schema drift | CI step generates the OpenAPI schema from the running FastAPI app and diffs against the committed schema snapshot — fails the build on undocumented drift |
