Both were written by the same session that's now naming this pattern for the fourth time; a self-check pass before invoking `/code-review` (already a standing CLAUDE.md step) didn't catch either, because both tests *looked* like they exercised the real mechanism at a skim — the gap only showed up when a reviewer asked "does this line actually call the function under test." Worth a sharper self-check heuristic, not just the existing one: for any test whose whole point is "does X actually get exercised," grep the test body for the literal call to X before trusting it, don't just read the assertion.

## 2026-07-15 (continued) — `/implement` Issue 02: ingest HTTP routes surfaced 5 real design gaps three risk-review rounds missed

**Context**: implemented `api-surface` Issue 02 (`POST /v1/ingest`, `GET /v1/ingest/{document_id}`) via `/implement`. The issue itself had already been through a full pre-publish gate, a post-publish independent re-check, and a full round of the user's own manual risk review (see the two entries above) — yet actually writing the code surfaced five more real gaps none of those passes caught: (1) `ingest_document()` can't accept a caller-supplied id, resolved by creating the job first and using its id as `document_id` too; (2) `SqlAlchemyACLLookup` requires a `document_versions` row that can't exist yet for a new upload — resolved with the user via a required `acl_override` form field; (3) `JobStore.fail()`'s raw error text would have crossed the HTTP trust boundary unredacted — both the field it stores in and the pipeline's own docstring had already flagged this for "whoever wires the route," but it still needed to actually be caught while wiring the route, not before; (4) auth has no admin-scope concept despite the spec requiring one, caught by `/code-review`'s Spec axis; (5) `corpus_id` is used for Qdrant collection naming but never defined anywhere, also caught by `/code-review`'s Spec axis.

**What worked well enough to repeat**:
- Reading the actual dependency signatures (`ingest_document`'s parameters, `SqlAlchemyACLLookup._fetch`'s query, `JobStore.create_job`'s return contract) before writing the route, rather than assuming "wire the pipeline to HTTP" meant a thin pass-through — a thin pass-through was structurally impossible here (no caller-supplied job id, no pre-existing ACL row), and both blockers were found by reading code, not by guessing and hitting a test failure later.
- Stopping to ask the user at the ACL-bootstrap gap rather than picking a default silently — matches the pattern from Issue 01's JWT decisions, and both times the "recommended, fail-closed" option was chosen, suggesting the framing (name the gap, offer 2-3 options, lead with the safest one) is landing correctly, not just procedurally followed.

**Friction / mistakes worth naming**:
- **The gap-finding didn't stop after three rounds of dedicated risk review — it just moved to a different phase.** Issues 01-04 collectively had a pre-publish gate, a post-publish independent re-check, and three rounds of the user's own domain-expert review before any code existed. None of those five new gaps were reachable from *reading the issue text* — every one of them only became visible once real code had to satisfy a real function signature (`ingest_document`'s parameter list) or a real query (`SqlAlchemyACLLookup._fetch`). This isn't a failure of the earlier review rounds; it's a real limit on what issue-text-level review can see at all, regardless of how many lenses or rounds are applied to it. Worth naming as its own category, distinct from "the review gate wasn't thorough enough": some gaps are structurally invisible until implementation, and the fix isn't more pre-implementation review, it's treating `/code-review` (which reads the actual diff against the actual spec) as a load-bearing, expected-to-find-real-things gate rather than a formality — which this session did, but worth stating as a deliberate policy rather than something that happened to work out twice.
