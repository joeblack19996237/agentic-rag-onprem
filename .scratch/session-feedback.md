## 2026-07-13 — CLAUDE.md/spec-tooling improvement pass (P0–P7)

**Context**: a long session implementing 8 concrete improvements against 2026 agentic-coding best practices (CLAUDE.md-as-router drift fixes, DEC-135 verification tiering, a doc-drift CI check, `docs/testing.md`, a pre-commit baseline, closing gaps in another session's cross-model `specs/` review, a handoff-doc convention + git tag, and moving/wiring up `docs/coding-standards.md`), plus this file.

**What worked well enough to repeat**:
- The loop of investigate → cite concrete evidence (exact file:line, not a general impression) → propose 2-4 scoped options → let the user pick one at a time → execute → verify by actually running it → report, repeated across every P-item. Keeping scope to one chosen item per turn kept each change reviewable and avoided speculative work nobody asked for.
- Actually running things caught real bugs that reading alone would have missed: `ruff-format` silently reformatting two unrelated files the first time it ran, the local `mypy` pre-commit hook needing an explicit `args: ["."]` (bare `pass_filenames: false` doesn't invoke it the way CI does), and the doc-drift checker's own regression proof — reproducing the exact hand-fixed README bug in a temp copy before trusting the tool, not just trusting its synthetic test fixtures.
- Not trusting a "this is already done" claim at face value: when told a cross-model review of `specs/` had already been triaged and fixed by a session with no visible history here, spot-checking the claimed fixes against real file content (not just reading the report) and running the mechanical safety nets surfaced two genuine incomplete-propagation gaps (a Drift Log row that stopped 3 findings into a 30-finding round, a missing Re-audit Note). The verification cost was small; skipping it would have left real gaps silently open.

**Friction / mistakes worth naming** (not promoted to standing rules yet — if any of these recur, that's the signal to):
- Appending `DEC-135` to `specs/13-decision-log.md` via an old_string/new_string edit anchored on the *start* of the existing last row (`DEC-134`) inserted the new row *before* it instead of after — broke append/chronological order. Caught only by a follow-up grep, not by the edit itself. When appending to an order-sensitive append-only log, anchor on the end of the current last entry, not the start of the row you don't want to precede.
- Nearly missed `.scratch/document-ingest-pipeline/PRD.md` when checking whether any in-flight `.scratch/` work cited DEC-135's changed content — the mental check defaulted to `.scratch/*/issues/*.md` (matching the issue-tracker doc's own convention) and didn't naturally extend one level up to a feature's PRD. Caught on a later pass, not the first one. "In-flight `.scratch/` work" means the whole feature directory, not just its `issues/` subfolder.
- Introducing a new tool for the first time on an established codebase (here: `ruff-format` via pre-commit) has a real chance of touching files outside the current task's scope, purely as a side effect of "nobody ran this formatter before." Worth specifically diffing what a first-time tool run touches, not just checking whether it passed.

## 2026-07-14 — Phase 2 kickoff: schema design, DEC-140/141, doc-drift fix, first Phase 2 issues published

**Context**: guided the user from "understand architecture + progress" through Phase 2 kickoff — discussed Postgres/Qdrant schema design for TASK-006/007, ran a full `verifiable-acceptance-criteria` gate (including a fresh independent sub-agent re-check) before publishing, found and corrected two real overclaims along the way, fixed a latent bug in the doc-drift checker itself, and published the first two Phase 2 issues (`.scratch/data-foundation/`).

**What worked well enough to repeat**:
- Actually running code to verify a technical claim, not just trusting a WebSearch summary, caught a real error the summary itself didn't lie about but was ambiguous enough to over-read charitably: DEC-140 claimed qdrant-client's local mode made payload-index checks "not guaranteed production-grade" (implying degraded-but-present); running it showed the index-creation call silently no-ops entirely (empty `payload_schema`, not observable at all). A verified *summary* of documented behavior is not the same as *running* the behavior in this repo's own environment — worth re-treating as a live check even after a citation is already confirmed live, not just after bare training-data recall.
- The independent fresh-subagent re-check (`verifiable-acceptance-criteria`'s Step 5) genuinely caught things the drafting context couldn't see on its own: the "hand-written, not autogenerated" AC has no after-the-fact verifying command (it's a provenance claim, not a checkable state) and the offline reversibility check proves structural symmetry, not the true round-trip-against-a-real-server claim TASK-006's own spec text asks for. Worth continuing to actually spawn this step rather than treating it as a formality once already confident.
- Following the append-only decision-log discipline even for a same-day, same-session correction (DEC-141 correcting DEC-140 minutes after writing it) rather than editing DEC-140 in place — kept the record honest about what was believed when, matching this project's own `audit_events` immutability rationale.
- Re-running `verifiable-acceptance-criteria` against the *actually-published* issue files (fresh `Read`, not conversation memory of the draft) when the user asked for a second pass caught a real thing the first pass missed entirely: `eval/` (one of Issue 01's own five owning modules) has a blanket import-graph ban that includes `config/`, contradicting the issue's own "What to build" instruction to put the shared SQLAlchemy Base in `config/`. The first pass's independent sub-agent check was still working from in-conversation draft text; the second pass's sub-agent was told to read real files and to actually reproduce claims (it dropped a probe file and ran the real checker) rather than reason about them abstractly — that instruction difference is likely why this one didn't surface earlier. Worth asking a re-check sub-agent to *reproduce*, not just *reason about*, a claim whenever the artifact it's reasoning about actually exists to be run against.

**Friction / mistakes worth naming** (not promoted to standing rules yet — if any of these recur, that's the signal to):
- Published two fresh, 0%-implemented issues with `Status: ready-for-human` instead of `ready-for-agent` — conflated "this issue will eventually need a human for its manual-verify tail" with the actual, narrower meaning this repo's own doc-drift checker hard-codes (`DONE_STATUSES = {"ready-for-human"}`): agent-closeable work is *already done*, only the human tail remains. The checker caught this immediately (a real, correct test failure), but it shouldn't have shipped in the first draft — re-read `docs/agents/triage-labels.md`'s literal definitions before assigning a Status line, don't infer meaning from the *shape* of a similar-looking precedent.
- The doc-drift checker's `find_issue_status_mismatches` keyed its status lookup by bare issue number across the entire `.scratch/` tree, silently colliding two different features' "Issue 01" — invisible until a second feature directory with its own from-01 numbering actually existed. A single-feature-directory codebase can't exercise this kind of cross-feature bug at all; a checker validated against only one instance of a repeating structure (one feature dir so far) hasn't actually been proven against the *repeating* case yet, even if every one of its unit tests passes.
- Did not append to this file, or check `tools/` for promotion-worthy scripts, until the user explicitly asked whether either had happened — despite `CLAUDE.md`'s own "Closing out a work cycle" checklist naming session-feedback as a step. The trigger ("before calling substantial work done") is a self-judged moment that never arrived in a long, continuously-flowing multi-topic session; it needs a harder, mechanically-detectable trigger (e.g. tied to the commit action itself) to not depend on the agent noticing on its own.

**Later the same day — implementing issues 01+02 (`/implement`, then `/code-review`)**:
- Running the actual `code-review` skill (parallel Standards + Spec sub-agents) after implementation caught 4 real issues a solo self-review missed, including a genuine type-fidelity bug (`audit_events.retrieved_chunk_ids` rendered `JSONB` instead of the spec'd `Array<String>`, inconsistent with how the identical declared type was handled correctly two tables earlier in the same migration file). None of the tests written *during* implementation happened to check those two specific fields' column types — worth remembering that "I wrote tests for what I built" and "an independent pass checks what I might have built wrong" are different guarantees, not redundant ones.
- The `verifiable-acceptance-criteria` gate had already run twice on Issue 02 (plus two independent sub-agent re-checks) before implementation started, and still missed a borrowed-artifact scoping error in AC4: it asserted on "the collection's payload field names," but Qdrant collections are schemaless until a point is written, and TASK-007 never writes one — that's TASK-008/009's job. All four prior gate passes checked "is there a command that produces this output" without checking the logically prior question, "does *this issue's own* deliverable actually produce the artifact the AC references." Caught only once implementation forced the question concretely (trying to write the test and finding there was nothing to assert against). Worth adding this as an explicit sub-question to the gate itself, not just relying on implementation to surface it.
- Adding `db` as a new import-graph module fixed one blanket-ban conflict (`eval/` needing `config/`) but silently created the identical conflict for every *other* blanket-ban module (`config` itself, `rerank`, `widget`) in the same instant, since all four share one `_all_other_modules()` helper. Caught only by immediately re-running the test suite after the addition, not by reasoning through the change's full blast radius first. When a shared helper is called from N sites, adding a new element to what it computes needs checking against all N call sites, not just the one that motivated the change.
