## What to build

A directly-callable ingest pipeline in `ingest/` covering `.md`/`.txt` documents end-to-end: format validation → chunking → embedding → Layer 1 ACL enrichment → Qdrant index write, with each step's output checkpointed so a job can be resumed later. The resume mechanism's own kill/restart chaos-test verification, and any format beyond `.md`/`.txt`, are out of scope here — see Issue 02 and `TASK-010`.

This is a callable pipeline entry point, not an HTTP route — `POST /v1/ingest` doesn't exist until `TASK-033` wires a route to this pipeline later; testing against an HTTP seam here would test something that doesn't exist yet.

Concretely:
- An entry point accepting file bytes + filename + `repository_id`, returning a job identifier. **Corrected during code review, 2026-07-15**: "immediately" describes the eventual HTTP-level behavior once `TASK-033` wires a route to this pipeline with real async/background dispatch — this issue's own callable (`ingest_document()`) runs the pipeline synchronously to completion (or failure) before returning, matching this issue's own scope boundary (the real asynchronous worker/dispatcher is `TASK-010`'s job, not this one's — building ad hoc threading here would duplicate or conflict with what `TASK-010` is meant to build properly). What this issue *does* guarantee is checkpointing: each step persists before the next starts, so a future caller (a real async dispatcher) can call this same function again for the same `job_id` and resume rather than restart. Any format other than `.md`/`.txt` gets the same rejection as a truly-unrecognized format — PDF/Word are not "supported, not yet implemented" at this slice, they're just rejected, until Issue 02 widens the accepted set.
- Chunking at 1024 tokens / 128-token overlap (recursive splitter primary, structural fallback), with `chunk_id` keyed by `(document_id, version_id, sequence)` and never reassigned once created. **Dependency gap found during `/verifiable-acceptance-criteria` (2026-07-14), sharpened during risk review (2026-07-15)**: no tokenizer library is pinned in `requirements.txt` yet, and the token count used for the 1024/128 split **must come from `bge-m3`'s own tokenizer** (XLM-RoBERTa-based, SentencePiece vocabulary via HuggingFace `transformers`/`tokenizers`, loading `BAAI/bge-m3`'s tokenizer config) — not character counting and not `tiktoken` (OpenAI's BPE vocabulary; a different tokenizer entirely, and character/tiktoken-based sizing produces chunk boundaries that drift from the intended token count and degrade sparse-vector feature quality). Picking and live-version-verifying the exact library (per this repo's dependency-version-claims rule) is a prerequisite for the `happy_path`/`chunk_immutability` ACs below.
- Embedding via `bge-m3`. **Architecture gap found during implementation, 2026-07-15**: TEI does not actually support bge-m3's sparse output — confirmed via TEI's own GitHub issue #141 (open): dense embedding works (bge-m3 is served as a regular XLM-RoBERTa model), but `/embed_sparse` is built for SPLADE-architecture models specifically, and bge-m3's own sparse mechanism needs the model's raw `last_hidden_state`, which TEI does not expose. This affects `specs/13-decision-log.md` DEC-035/DEC-086/REQ-003 beyond this issue's own scope — `ingest/embedding.py`'s real `TEIEmbeddingClient` is built honestly (dense via the confirmed-working `/embed` call; the full dense+sparse method raises `TeiEmbeddingUnsupportedSparseError` rather than fabricate a sparse vector), not silently worked around. Needs a `DEC-###` entry and a real decision (e.g. serving bge-m3's sparse side via BAAI's own `FlagEmbedding` library directly) before ingest can be wired to a live embedding service — this issue's own ACs are unaffected since they're all fake-backed (Tier-1, DEC-135). A new small query in `config/` (`get_active_model_version`) reads the active `embedding`-role `model_version`, wired into a new `ingest/qdrant_setup.py::resolve_index_target()` that combines it with `build_collection_name` to produce the actual `IndexTarget` a caller passes to the pipeline — added during code review, 2026-07-15, after the query was found built but never actually called from anywhere outside its own test.
- A new, deliberately minimal `acl/` stub: two functions (an effective-ACL lookup, a retention-state lookup) reading `document_versions` directly. This is *not* a forward-designed adapter interface for `TASK-013` to reuse — `TASK-013` designs the real Layer-2 `acl/` adapter independently in Phase 3 and may replace this stub outright. Before a chunk is written to the index, its payload gets `allow_principals[]`, `deny_principals[]`, `security_label`, `retention_state` from this stub; if the lookup fails, ingest blocks entirely rather than writing an ACL-less chunk (fail-closed, not fail-open).
- A new job-store seam: a narrow port (create job, read phase/payload, advance phase/payload) with two implementations — a real SQLAlchemy/Postgres one against the existing `job_queue` table, and an in-memory fake for tests. `job_queue.status` (the real `CHECK (status IN ('pending','in_progress','complete','failed'))` column, already migrated by TASK-006) is generic dispatch bookkeeping shared with CDC jobs — it is not the ingest-specific state machine. The `pending → parsing → indexing → ready` phase and a progress fraction live inside `job_queue.payload` (JSONB), read/written through this port; the port is also the place any other step-level flag gets recorded (e.g. Issue 02's `parser_fallback`) — `job_queue.payload` has no fixed-field contract, unlike Qdrant's chunk payload (NFR-003), so it can carry whatever small application-level state a step needs without a spec change.
  **`job_queue.payload` is lightweight bookkeeping only — never the bulk step output itself** (clarified during risk review, 2026-07-15; the original wording — "each step persists its output... into `job_queue.payload`" — was ambiguous enough to read as "store the actual parsed text/chunks in Postgres JSONB," which is not the intent). A 100-page PDF's parsed text can run to several MB; repeatedly `UPDATE`-ing that much data into a JSONB column on every step would create real Postgres WAL write amplification and TOAST pressure under load. Instead, each step writes its bulk output (parsed text, chunk list, embeddings) to a checkpoint file on a local host-mounted volume (`./ingest_checkpoints/`, the same pattern `specs/07-database.md`'s Postgres/Qdrant volumes already use — no new infrastructure, no object storage), and `job_queue.payload` stores only a small pointer (e.g. `{"phase": "chunked", "progress": 0.4, "checkpoint_path": "./ingest_checkpoints/<job_id>-chunks.json"}`) plus small flags. Each step persists its checkpoint file and advances the pointer/phase before the next step starts; re-invoking the pipeline against a `job_id` whose earlier steps already completed reads the checkpoint file rather than redoing them.
- Identity/ACL fields never enter the text sent to the embedding service (NFR-012).
