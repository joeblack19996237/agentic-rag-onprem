# PRD: Document Ingest Pipeline (Parse, Chunk, Embed, Index)

## Problem Statement

An enterprise admin or a CCM/ECM vendor's automated sync process has a document that needs to become queryable. Today there is no ingest path at all: no way to submit a document, no way to know whether it parsed, no way to know when it is safe to query. Until a document is parsed, chunked, embedded, and indexed with its access-control data attached, it cannot be found by any query, and if it were indexed without its ACL data, it would leak content to users who should not see it.

## Solution

A pipeline that parses a supported-format document, splits it into fixed-size overlapping chunks, embeds each chunk with a dense+sparse embedding model, attaches Layer 1 access-control and retention metadata to each chunk's payload, and writes the result into the vector index (built by TASK-007, already exists as `ingest/qdrant_setup.py`). Progress moves through `pending → parsing → indexing → ready` (or `failed`), persisted so a caller can check status and so a future resume mechanism can pick up from the last completed step. **Clarified during Issue 01 code review, 2026-07-15**: "asynchronous" describes the eventual product behavior once a real dispatcher exists — this PRD's own pipeline callable runs synchronously to completion when invoked; genuine background/async execution (a worker claiming queued jobs, e.g. via `SELECT ... FOR UPDATE SKIP LOCKED`) is `TASK-010`'s scope, not built here. What this PRD's scope guarantees is the checkpointing that makes a *future* async dispatcher's resume-after-restart behavior possible, not the dispatcher itself.

This PRD delivers the pipeline as a **directly-callable module**, not yet reachable over HTTP. `POST /v1/ingest` / `GET /v1/ingest/{document_id}` are specified in `specs/06-api-contracts.md` §2, but registering those FastAPI routes and wiring them to this pipeline is `TASK-033`'s job (depends on `TASK-006`, `TASK-009` — i.e. runs after this PRD), confirmed by `api/main.py` today having only `GET /ready`. Testing this PRD against an HTTP seam would test a route that doesn't exist yet.

## User Stories

1. As an enterprise admin, I want to upload a PDF/Word/Markdown/plain-text document, so that its content becomes queryable by my organization's users.
2. As an enterprise admin, I want an immediate `document_id` and a way to check status on upload, so that I don't have to block on a long-running call to know my upload was accepted.
3. As an enterprise admin, I want to poll ingest status, so that I know when a document is safely queryable versus still processing.
4. As an enterprise admin, I want an upload in an unsupported format to be rejected immediately with a clear list of supported formats, so that I don't wait for a job that can never succeed.
5. As a CCM/ECM vendor integrator, I want to trigger ingest via an automated sync process (not just manual upload), so that documents already living in the host ECM/CCM system stay in sync without a human copy-pasting files.
6. As an enterprise admin, I want every chunk of a document to carry the same access-control list as the source document at the time of ingest, so that a user without permission to see the source document can never see it via a query answer.
7. As an enterprise admin, I want a document's retention state captured at ingest time, so that documents under legal hold or past their retention window are excluded from future answers even if the retrieval-time ECM check lags.
8. As an enterprise admin, I want ingest failures (unreachable embedding service, unreachable ECM ACL lookup) to retry rather than silently drop the document, so that a transient outage doesn't produce a permanently un-ingested document with no visible error.
9. As an enterprise admin, I want each pipeline step's output persisted before the next step starts, so that a future resume mechanism (`TASK-010`) can restart an interrupted job from its last completed step instead of from scratch.
10. As an end user querying the system, I want documents that finished ingest to be retrievable within their normal query latency budget, so that a newly indexed document behaves like any other document, not a second-class one.
11. As an enterprise admin, I want a large document (up to the reference 100-page benchmark) to reach `ready` status within a bounded time (≤ 60s on reference hardware), so that ingest throughput is predictable and demoable.
12. As a system operator, I want it to be structurally impossible for a chunk to be written to the index without its ACL payload attached, so that a race condition or partial failure cannot produce a universally-visible chunk (fail-closed, not fail-open).

## Implementation Decisions

- **Seam (corrected from the 2026-07-13 draft)**: the external seam this PRD tests is a direct Python callable in `ingest/` — an entry point that accepts a parsed upload (file bytes, filename, `repository_id`) and returns a job handle, plus a status-read function. `POST /v1/ingest` / `GET /v1/ingest/{document_id}` are `TASK-033`'s wiring, layered on top of this callable later; that wiring is out of scope here (see below). Re-uploading the same file content produces a new `document_id`; content-level idempotency is the caller's responsibility, not this pipeline's.
- **Format support and rejection**: supported formats are PDF (text-extractable), Word (`.docx`), Markdown, and plain text. An unsupported format is rejected synchronously (as a raised exception or a typed error result — `ingest/` returns a domain-level rejection, it does not itself construct an HTTP 415; `TASK-033` maps it to the wire-level 415), before any job row is created.
- **Parsing**: Unstructured.io is the primary parser; PyMuPDF is the rescue path for PDF layouts Unstructured mangles; `python-docx` handles Word. OCR and complex-table extraction are explicitly deferred (out of scope below).
- **Chunking**: Fixed chunk size of 1024 tokens with 128-token overlap. Recursive splitter is the primary strategy; a structural splitter (markdown headers, PDF sections) is the fallback. `chunk_id` is immutable, keyed by `(document_id, version_id, sequence)` — this identity must never be reassigned once a chunk is created, since citation and audit records depend on it staying stable.
- **Embedding — corrected 2026-07-15, DEC-142**: `bge-m3` served via TEI produces the dense vector; a separate, dedicated SPLADE-architecture model, also served via TEI in its own deployment, produces the sparse vector — TEI cannot serve `bge-m3`'s own sparse output (verified against TEI's actual API during Issue 01's implementation, not the "single forward pass produces both" framing this line originally carried). This pipeline is what makes REQ-003's "hybrid dense+sparse" retrieval possible at all — it is the only place either vector is produced, now from two models instead of one — but REQ-003's own NDCG/rerank acceptance criterion is exercised at query time (Phase 3), not tested here. See `specs/13-decision-log.md` DEC-142 for the full rationale and accepted trade-offs (weaker dense/sparse complementarity than one jointly-trained model would give; two embedding calls instead of one, at query time too, not just ingest; an unresolved VRAM-budget addition to `04-architecture.md` §4.2.2).
- **Active embedding model version**: read from `config/model_versions` (already modeled in `config/models.py`, `ModelVersion.is_active` partial-unique per role) — this pipeline adds the runtime query (`role='embedding', is_active=true`) needed to pick the target Qdrant collection name via `ingest/qdrant_setup.py`'s existing `build_collection_name(corpus_id, embedding_model_version)`. No such query helper exists yet; this PRD adds it.
- **ACL enrichment (Layer 1) — corrected against actual repo state**: `acl/` currently contains only a placeholder `__init__.py` (verified 2026-07-14); no `ECMAdapter`/`LocalAdapter` code exists yet — the real Layer 2 JIT adapter (`batch_check_access`, PDP circuit breaker, and whatever shape `ECMAdapter` ultimately takes) is `TASK-013`'s Phase 3 deliverable, designed then, not now. This PRD does **not** attempt to pre-build that interface or promise `TASK-013` will extend it. Instead it adds a minimal, throwaway lookup used only by this pipeline — two functions (an effective-ACL lookup and a retention-state lookup) reading straight from `document_versions` (the same Postgres row `05-data-model.md` already names as the no-ECM source of truth) — with no claim about being "the" adapter interface. `TASK-013` is free to redesign `acl/` from scratch when it actually gets built; this stub may be reused, wrapped, or deleted outright at that point. Before a chunk is written to the index, its payload is enriched with `allow_principals[]`, `deny_principals[]`, `security_label`, and `retention_state` sourced from this stub. This is Layer 1 of the two-layer authorization model; Layer 2 (live re-check at query time) is out of scope for this pipeline.
- **Identity must never enter the embedding vector.** ACL and identity fields are payload-only metadata alongside the vector, never part of the text that gets embedded (NFR-012). This is a hard constraint, not a style preference — an embedding that encodes identity could leak access information through vector similarity itself.
- **Index write**: Qdrant collection, keyed by `<corpus_id>_<embedding_model_version>` (`ingest/qdrant_setup.py`, already built by TASK-007). A chunk is only considered indexed once its vector and full ACL/retention payload are written together — partial writes (vector without ACL payload, or vice versa) must not be observable as a retrievable state.
- **Job durability and resume — corrected against the real schema**: Ingest jobs live in the real `job_queue` table (`ingest/models.py`, built by TASK-006). Its `status` column is a **generic dispatch-lifecycle value** (`CHECK (status IN ('pending', 'in_progress', 'complete', 'failed'))`), shared with CDC re-poll jobs per `specs/07-database.md` — it is **not** the same state machine as the ingest-specific `pending → parsing → indexing → ready` progression the API contract exposes. This PRD's own scope must place that finer-grained phase inside `job_queue.payload` (JSONB) — e.g. a sub-field tracking the current ingest phase and a progress fraction — read back and translated to the contract's `{status, progress, errors}` shape by whichever later code exposes it over HTTP (`TASK-033`). **`job_queue.payload` is lightweight bookkeeping only, never the bulk step output** (clarified during risk review, 2026-07-15 — the original wording here was ambiguous enough to misread as "store the parsed text/chunks directly in Postgres JSONB," which risks real WAL/TOAST write amplification on a multi-MB document; see Issue 01 for the corrected design — bulk output goes to a local checkpoint-file volume, `job_queue.payload` holds only a pointer). Each pipeline step (parse+chunk, embed+ACL+index — whether chunking is its own checkpoint or bundled with parse is an implementation-time call) persists its output to a checkpoint file and advances the payload-level phase/pointer before the next step starts, and re-running an already-completed step against the same `job_id` must be idempotent (same result, not a duplicate) — this is the structural property `TASK-010`'s later chaos test (kill mid-embed, confirm resume) actually exercises; this PRD is responsible for making that property true and directly assertable, not for the kill/restart chaos test itself.
- **Job-store seam (found 2026-07-14, verified empirically)**: `job_queue.payload` is `postgresql.JSONB` and `documents.approver_user_ids` is `postgresql.ARRAY` — confirmed by actually running `Base.metadata.create_all()` against an in-memory SQLite engine, which fails to compile both types (`CompileError: can't render element of type JSONB` / `ARRAY`). Unlike Qdrant, this stack has no embedded-local-mode escape hatch for Postgres, and this sandbox has no Docker to run a real one (`docs/agents/dev-environment.md`'s verification ceiling). So the pipeline's checkpoint/idempotency logic must sit behind a small **job-store seam** — a narrow port (create job, read phase/payload, advance phase/payload) with a real SQLAlchemy/Postgres implementation and an in-memory fake for tests — the same Protocol-plus-fake pattern as the ACL stub above, not a new one invented for this. The fake makes the pipeline's own checkpoint/idempotency behavior Tier-1/agent-executable; the real Postgres-backed implementation's runtime correctness (does the actual `UPDATE ... SET payload = ...` behave as expected against a live server) is `[manual-verify]`, same tier as the existing Qdrant index-existence gap from DEC-140/141.
- **Failure handling**: If the embedding service is unreachable, the job requeues with backoff and remains visible at `status = pending` with an ops-visible retry count — it must not silently hang. If the ECM ACL lookup is unreachable, ingest blocks entirely rather than defaulting to an unrestricted/no-ACL payload; a chunk must never be written to the index without its ACL payload, even under a dependency outage.
- **Modules touched**: `ingest/` (parse, chunk, embed, ACL enrichment, index-write steps, job-store port + real/fake implementations — `qdrant_setup.py` already exists, models already exist), `acl/` (new: a minimal, throwaway ACL-lookup stub scoped only to this pipeline's own two calls, not a forward-designed Protocol — this is a real, small addition, not the read-only consumption the 2026-07-13 draft assumed), `config/` (new: the active-embedding-model-version query helper).

## Testing Decisions

- Tests target the pipeline's directly-callable entry point and observable index/queue state — not the internal parse/chunk/embed functions in isolation, and not the `POST /v1/ingest` HTTP layer (that doesn't exist until `TASK-033`). Only external behavior (job status transitions, final index content and payload shape) is asserted.
- TDD Red for the parse+chunk step: calling the pipeline entry point with a supported-format document returns a `document_id` and an initial `pending` phase; calling it with an unsupported format raises/returns the typed rejection with no `document_id` issued.
- TDD Red for the embed+enrich+index step: a chunk's embedding input never contains an ACL/identity field (this is a static/CI-enforceable check, not just a runtime assertion); a chunk's Qdrant payload contains `allow_principals[]`, `deny_principals[]`, `security_label`, `retention_state` correctly populated. Use the new narrow `LocalAdapter` (no live ECM) for this test.
- Acceptance test: end-to-end ingest of a sample 100-page PDF reaches the pipeline's `ready` phase and its chunks are retrievable via a direct Qdrant query, within the reference-hardware timing budget.
- **Carried from the 2026-07-13 draft, found implementing `data-foundation` Issue 02**: NFR-003's full schema-neutrality check (the written chunk payload's field-name set matches `specs/07-database.md`'s documented 12-field list exactly) belongs here, not on TASK-007's collection-setup issue — Qdrant collections are schemaless until a point is written, and TASK-007 never writes one. Include this as an acceptance criterion on whichever slice actually writes the chunk payload.
- The NFR-012 "no identity in embedding vector" check must be enforceable in CI as a static check, not only as a per-test runtime assertion — a regression here is a security defect, not a functional one.
- Chunk immutability (`chunk_id` stability per `(document_id, version_id, sequence)`) should be asserted directly, since citation/audit correctness in later pipeline stages depends on it.
- Step-checkpoint idempotency (re-invoking the pipeline against a `job_id` whose first N steps already persisted output resumes rather than reruns them) should be asserted directly at this Tier-1/agent-executable level, against the in-memory job-store fake (see the "Job-store seam" implementation decision) — this is distinct from `TASK-010`'s actual kill/restart chaos test, which is `[manual-verify]` (see Out of Scope).
- **Job-store real-implementation gap**: the fake job-store proves the pipeline's own checkpoint logic is correct; it does not prove the real SQLAlchemy/Postgres-backed implementation of the same port behaves identically against a live server (JSONB read/write round-tripping, concurrent `UPDATE` semantics). That is `[manual-verify]`, owner Backend, evidence = running the pipeline against a real `docker compose up` Postgres and confirming the same test scenarios pass — do not let this gap get silently assumed-covered by the fake's green tests.
- **Carried from the 2026-07-13 draft, `specs/13-decision-log.md` DEC-140/DEC-141**: the "retrievable via a direct Qdrant query" part of the acceptance test's retrieval/filter behavior should run against a `QdrantClient(":memory:")` instance directly (real payload write + real filter query) — genuinely agent-executable. Payload-index *existence* is a separate, still-Tier-3/`[manual-verify]` concern (`create_payload_index()` silently no-ops under local mode) — already handled by TASK-007's own issue, not re-tested here.
- Prior art: `tests/ingest/test_qdrant_setup.py` (TASK-007) for the `QdrantClient(":memory:")` pattern; `tests/db/test_migration.py` (TASK-006) for the `job_queue`/`documents`/`document_versions` schema this pipeline writes against. `specs/11-test-plan.md`'s `TEST-###` catalog should be consulted for the canonical test IDs this PRD's tests map to before implementation starts.

## Out of Scope

- **Registering `POST /v1/ingest` / `GET /v1/ingest/{document_id}` as actual FastAPI routes** — this is `TASK-033`'s scope (depends on this PRD's `TASK-009` completing first); this PRD only needs to expose a callable entry point `TASK-033` can wire up.
- **The kill/restart chaos-test verification of resume** — `TASK-010`'s scope. This PRD is responsible for the checkpoint-persistence mechanism being structurally correct and idempotent (see Testing Decisions); actually killing and restarting a live process to prove it is `TASK-010`'s `[manual-verify]`-flavored TDD-Exempt task.
- **Layer 2 JIT authorization** (`batch_check_access`, PDP circuit breaker, live re-check at query time) — `TASK-013`'s scope, Phase 3. This PRD's `acl/` addition is a disposable stub limited to the two Layer-1 lookups named above; it makes no claim about `TASK-013`'s eventual design.
- OCR and complex-table extraction (deferred to a later phase, per DEC-036).
- CDC-driven re-ingest on document version changes (webhook or poll-based sync from the host ECM) — covered by separate build-plan tasks (CDC consumer tasks), not this PRD.
- The `GET /v1/ingest/{document_id}` response's full status-object schema beyond the state machine itself (exact JSON shape is defined in `specs/06-api-contracts.md` §2 and should be implemented as specified there, not re-derived here, once `TASK-033` wires it up).
- Legal-hold freeze and cache invalidation behavior for already-indexed chunks — covered separately (retention/legal-hold build-plan tasks).
- Multi-host / distributed job queue (Postgres `SELECT FOR UPDATE SKIP LOCKED` is explicitly a single-host MVP/V2 pattern; V3 multi-host may reconsider per DEC-038).

## Further Notes

- **What changed from the 2026-07-13 draft, and why**: this version was regenerated via `/to-prd` rather than patched, because `specs/` had changed since the original draft (DEC-140/141) and because TASK-006/TASK-007 — which the original draft could only reason about as future dependencies — are now real, running code. Re-reading `TASK-033`'s actual text, `api/main.py`'s actual current content, `ingest/models.py`'s actual `job_queue.status` CHECK constraint, and `acl/`'s actual (empty) directory — plus actually running `Base.metadata.create_all()` against SQLite — surfaced four concrete corrections the original draft could not have made from prose alone:
  1. The original draft's seam (`POST /v1/ingest`) is `TASK-033`'s deliverable, not this PRD's — `api/main.py` today has only `GET /ready`.
  2. `job_queue.status`'s real CHECK constraint (`pending/in_progress/complete/failed`) is coarser than the ingest-specific `pending/parsing/indexing/ready` progression the API contract describes — the finer phase must live in `job_queue.payload`, a decision the original draft didn't have to make because the table didn't exist yet.
  3. `acl/` has no adapter code at all yet (`TASK-013` is Phase 3) — this PRD adds a minimal, throwaway ACL-lookup stub of its own, not a forward-designed `ECMAdapter` Protocol; this is a real modification, not the "read-only consumed" framing the original draft assumed. `TASK-013` designs the real adapter independently when it lands.
  4. `job_queue.payload` (JSONB) and `documents.approver_user_ids` (ARRAY) don't compile against SQLite (verified by actually running it) — unlike Qdrant, there's no embedded-local-mode escape hatch for Postgres in this sandbox, so this PRD adds a small job-store port + fake specifically to keep checkpoint/idempotency logic testable here, with the real Postgres-backed implementation flagged `[manual-verify]`.
- This PRD still merges `TASK-008`/`TASK-009` into one vertical slice for the same reason as before: they share one seam (now: the pipeline callable, not the HTTP route) and neither is independently demoable — a document that only reaches "parsed and chunked" but not "embedded and indexed" is not yet queryable.

The new `acl/` stub and job-store port are both small but add scope the estimate didn't originally price in.
