## What to build

`GET /v1/admin/audit` (paginated read over `audit/models.py`'s `audit_events` table) and `GET /v1/admin/config/models` (active `model_versions` row per role).

**Audit list.** `audit_events` exists as a real, migrated table (`data-foundation` TASK-006) but nothing in this codebase writes to it yet — no code anywhere constructs an `AuditEvent(...)` row outside the model file itself (confirmed by grep while drafting this issue). `TASK-024` (not built) is what eventually writes real rows during query handling. This issue's own scope is the *read* side only: given rows seeded directly, `GET /v1/admin/audit` returns them correctly filtered (`from`, `to`, `user_id`) and cursor-paginated. Don't wait on `TASK-024` — a read endpoint over an existing (if currently empty in production) schema doesn't need a producer to already exist to be correctly built and tested, the same way `document-ingest-pipeline`'s tests never needed a live Postgres to prove `job_queue` read/write logic. **Indexes, checked during risk review**: `audit_events` already has `ix_audit_events_user_timestamp` (`user_id, timestamp` composite) and `ix_audit_events_timestamp`, both from `data-foundation` TASK-006's real migration — the `user_id`/`from`/`to` filter combination this AC needs is already index-backed, not a gap this issue needs to close. **`user_id` format, clarified during risk review**: `AuditEvent.user_id` is a plain `String` column with no format constraint or FK — this issue doesn't own the write side (`TASK-024`), so it doesn't invent a normalization scheme; the filter is an exact string match against whatever `audit_events.user_id` actually contains, no case-folding, no assumed structure.

**Model-version read.** Extends `config/active_model_version.py`'s existing `get_active_model_version(session, role=...)` (built for the ingest pipeline's own need) into a list-all-roles read: `generation`, `embedding`, `rerank`, `nli`, `safety_input`, `safety_output`, `policy` (the seven roles named in that module's own `NoActiveModelVersionError` docstring). No new query pattern, just a loop over the known role set reusing the existing single-role query. **Hardcoded role list, defended during risk review**: don't derive this list dynamically (e.g. `SELECT DISTINCT role FROM model_versions`) — that would silently omit a role that was never seeded (no active row ever written), which is exactly the broken state an admin diagnostic endpoint should surface, not hide. The 7-role list is already this codebase's own fixed convention (`config/active_model_version.py`'s own docstring names the same 7, per `specs/07-database.md`'s Seed Data section), not a new decision invented here. **Missing-active-version handling, resolved during risk review**: `get_active_model_version` raises `NoActiveModelVersionError` for a role with no active row (a real misconfiguration, not a normal state). Catch it per-role rather than letting one broken role fail the whole request: return `{role, model_version: null}` for that role and log a warning, so the rest of the response still succeeds. **Multiple-active-rows, checked during risk review, not actually an open question**: `get_active_model_version`'s existing `.scalar_one_or_none()` call already raises `sqlalchemy.exc.MultipleResultsFound` if more than one active row matches a role (confirmed empirically against this repo's SQLAlchemy install) — and `model_versions` has a real partial unique index (`ix_model_versions_one_active_per_role`, `WHERE is_active = true`) making "two active rows for one role" structurally impossible in Postgres in the first place. Nothing new to design here.

**Response shape, resolved during the post-publish `/verifiable-acceptance-criteria` re-check**: `06-api-contracts.md`'s own wording ("Active `model_versions` per role") doesn't say whether the response is the full DB row or just the version — and `get_active_model_version` itself only returns the bare `model_version` string, not a row. Return `{role, model_version}` pairs for each of the 7 roles (extending the existing function's return type with the role it was queried for) — not the full `model_versions` row (`adapter_name`, `is_active`, timestamps, etc. are internal bookkeeping this read-only admin surface doesn't need to expose).

**Testing tiers, made explicit during risk review** — same split as Issue 03, applied here: **Tier A** (mocked `Session` + compiled query, for AC1 and AC3) proves SQL construction correctness for a single call; **Tier B** (a new `FakeAuditEventStore`, in-memory and dict-backed, same shape as `ingest/job_store.py`'s `InMemoryJobStore` — for AC2) proves the pagination algorithm against known, multi-call data. Deliberately two separate fakes (`FakeDocumentStore` from Issue 03, `FakeAuditEventStore` here), not merged into one generic pagination-test helper — two occurrences isn't yet a pattern worth abstracting; if a third paginated admin list shows up later, that's the trigger to extract a shared helper, not this issue. Neither tier, nor their combination, proves genuine Postgres MVCC concurrent-transaction behavior — that gap is Manual Verification's job below, not silently assumed closed.
