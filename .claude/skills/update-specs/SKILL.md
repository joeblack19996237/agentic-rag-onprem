# Update Specs

Keep `specs/` internally consistent, fully cross-referenced, and traceable into `.scratch/` PRDs and issues every time its content changes.

## Why this discipline exists

`specs/` was generated by `idea-to-specs` through a multi-stage pipeline with a stable ID system (`REQ-###`, `NFR-###`, `RISK-###`, `DEC-###`, `TASK-###`, `TEST-###`, `VG-###`) and a final quality-audit gate. Every file cross-references others by these IDs, not by prose restatement — that's what makes the package machine-checkable instead of just readable. `specs/00-index.md` is the map that tells a reader (human or agent) what's where; `specs/13-decision-log.md` is the canonical, append-only decision record; `specs/14-spec-audit-report.md` is the last audit verdict.

None of that is self-maintaining. An edit that changes a REQ's meaning without touching the build-plan task that implements it, or a new decision that never gets an ID, or a `READY` verdict left standing over content nobody re-checked — these are exactly how a spec package quietly drifts from "trustworthy source of truth" into "a pile of markdown files that used to be accurate." This skill exists to make each edit propagate to everywhere it needs to, in one pass, instead of relying on remembering to come back later.

## Step 0 — Classify the change

Two kinds of change, with different procedures below:

- **Procedure A — New spec slot.** A whole new file under `specs/`: a previously-skipped slot gets activated, a new conditional slot turns on, or an existing slot gets a real successor.
- **Procedure B — Amend existing content.** A REQ/NFR/DEC changes, an existing section gets deepened, a design gap gets closed inside a file that already exists.

Some changes are both — a new file that also revises an existing REQ. Do both procedures in that case, in the order they appear below (Procedure A creates the file and its slot; Procedure B's cross-reference and re-audit steps still apply to whatever existing content the new file touches).

## Step 1 (both procedures) — Map the blast radius before editing

Before changing anything, find every file that already references the ids or topic you're about to touch:

```
grep -rn "<ID-or-topic>" specs/
grep -rn "<ID-or-topic>" .scratch/
```

Do this even when you're confident you know where the references are — memory of a 500KB spec package is unreliable, and the whole point of this step is to catch the reference you forgot about. This is the single step most likely to get skipped under time pressure, and skipping it is the most common way specs actually drift: not from bad edits, but from good edits that stopped one file short of where the change actually reaches.

**Grep the underlying entity, not just the formal ID.** A requirement or decision ID (`NFR-026`, `DEC-016`) only surfaces the files that cite that ID by name — it misses files that discuss the same underlying thing in its own vocabulary. If the change touches a data entity, module, or concept with a name of its own (e.g. `otel_spans`, `chunks`, `acl/`), grep that name too. In practice this is often what actually finds the schema-layer files (`specs/05-data-model.md`, `specs/07-database.md`) — they describe the entity directly and may never mention the requirement ID that motivated the change at all.

## Step 2 (both procedures) — Mint new IDs, never reuse or renumber

Each ID namespace has one canonical file and lives in `specs/13-decision-log.md` (`DEC-###`) or a `spec-writer`-owned file. Find the next free number in the relevant namespace before writing anything:

```
grep -oE "<PREFIX>-[0-9]+" specs/<canonical-file>.md | sort -t- -k2 -n -u | tail -1
```

Never renumber or repurpose an existing ID to make room for a new one — every existing cross-reference to that ID would silently point at the wrong thing. Skipped slots stay skipped; superseded decisions stay in the log with a note, not deleted.

| Namespace | Canonical file | Notes |
|---|---|---|
| `REQ-###`, `NFR-###`, `RISK-###` | `specs/02-requirements.md` | |
| `DEC-###` | `specs/13-decision-log.md` | Append-only. If any other file disagrees with this file, the decision log wins and the other file is stale — fix the other file, not this one. |
| `TASK-###` | `specs/10-build-plan.md` | |
| `TEST-###` | `specs/11-test-plan.md` | |
| `VG-###` | `specs/12-verification.md` | |
| Slot numbers (`00`, `01`, `02`, … `93`, plus the 20s/30s/40s/50s conditional families) | `specs/00-index.md` registry | Stable slot map lives in `.claude/skills/idea-to-specs/references/spec-catalog.md` |

## Procedure A — Register a new spec slot

1. **Assign the slot number** from the stable slot map in `.claude/skills/idea-to-specs/references/spec-catalog.md`. Never renumber an existing slot to make room.
2. **Add the file to the matching review group** in `specs/00-index.md`'s Review Groups section (A–E, or a new group letter if it doesn't fit an existing theme). A file with no group assignment is invisible to anyone scoping a review by group.
3. **Move it out of the Skipped table** in `specs/00-index.md` if it was previously listed there, and into the Selected/Generated table with slot, file, size, key IDs introduced, and what it traces to.
4. **Cross-check new IDs** minted by the new file don't collide with existing ones (Step 2 above), and that any `REQ`/`NFR`/`RISK`/`DEC` IDs it mints are mirrored into their canonical files.
5. **Propagate to `CLAUDE.md` and `docs/agents/domain.md`** if the new file belongs in the always-read core set (most product-level specs do) — those two files are what downstream skills (`to-prd`, `to-issues`, `implement`, `triage`) actually read; an update to `00-index.md` alone does not make a new file discoverable to them.
6. Continue into Procedure B's steps 3 onward for the content itself (upstream-first ordering, decision log, re-audit, `.scratch/` check) — a new file is still content that needs the same propagation discipline.

## Procedure B — Amend existing spec content

1. Step 0/1/2 above are done: change classified, blast radius mapped, new IDs minted if needed.
2. **Edit upstream before downstream.** Requirements (`REQ`/`NFR`) and decisions (`DEC`) are the upstream layer — edit those first. Everything else (architecture prose, build-plan tasks, test/verification IDs) should trace back to them, not the other way around. Editing downstream first risks writing prose that a later-added requirement then contradicts.
3. **Append to the decision log, never rewrite a row.** Add a new `DEC-###` row in `specs/13-decision-log.md`; if it changes an earlier decision's effect, say so in the new row's text ("supersedes DEC-016's implicit default for X") rather than editing the old row. This mirrors the product's own `audit_events` design principle elsewhere in these specs — append-only history is the pattern this project already trusts for exactly this reason.
4. **Propagate the new IDs into `specs/00-index.md`.** Update the "Key ids" column for every file you touched, and give a file a Group assignment if it didn't have one before. Skipping this step is how the index goes stale — and a stale index is worse than no index, because people trust it without checking.
5. **Re-audit `specs/14-spec-audit-report.md` — don't leave a stale verdict standing.** This file already has a precedent for how to do this: look at its existing "Re-audit Note" entries and follow the same pattern — append a dated note naming exactly which gates are affected by the new content, re-run only those, and explicitly carry forward the rest as still-PASS from the original audit. A `READY` verdict that was never actually re-checked against new content is a false signal to everyone downstream who trusts it.
6. **Check `.scratch/` for anything already published that cites what you just changed.**
   ```
   grep -rln "<changed-IDs>" .scratch/
   ```
   For each match: if the referencing PRD or issue hasn't been implemented yet, update its citations (Implementation Decisions, Acceptance Criteria, Blocked-by) to match the new spec content, and note the update under its "Further Notes" section. If it has already been implemented, don't silently edit history — flag the discrepancy as a new, separate follow-up instead, since the fix now lives in code, not in a doc.
7. **Confirm the root `CLAUDE.md` supersedes rule is satisfied.** It requires: a supersedes entry in `13-decision-log.md` (step 3), a check of whether `docs/adr/` needs updating (check even if the directory doesn't exist yet — the rule still applies once it does), and verification of in-flight `.scratch/` features (step 6). Treat this as the final gate, not a separate parallel process — steps 3–6 above are what satisfy it.

## Quick reference — what else to check, by change type

| If you're changing... | Also check |
|---|---|
| A `REQ` or `NFR` | The `TASK-###` in `specs/10-build-plan.md` that implements it — does its Acceptance Criteria still match? `TEST-###`/`VG-###` coverage in `specs/11-test-plan.md`/`specs/12-verification.md`. The owning file's "Key ids" row in `specs/00-index.md`. |
| A `DEC` | Every file listed in that decision's "Dependencies"/"Decision References" footer — grep to confirm that list is actually exhaustive, don't just trust it. Whether it supersedes an earlier `DEC` (say so explicitly in the new row, per Procedure B step 3). |
| An architecture section (`specs/04-architecture.md`) | The narrower spec file that expands it (e.g. an observability sketch in `04-architecture.md` expands into `specs/08-observability-logs.md`). Whichever `TASK-###` implements that section. `specs/00-index.md`'s Architecture review matrix offsets, if line numbers shifted. |
| Anything with a matching `TASK-###` already turned into a PRD or issue | Grep `.scratch/` for that `TASK-###` and every `REQ`/`NFR`/`DEC` it cites (Procedure B step 6). |
| Anything at all | Whether `specs/14-spec-audit-report.md`'s `READY` verdict now needs a Re-audit Note (Procedure B step 5). |
